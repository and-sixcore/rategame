# Workflow

How a feature moves from a blank frame to something engineering can build. Take a new
games screen as the running example.

<Steps>
  <Step title="Explore in Workboard">
    Build the new games screen in **Workboard**, the no judgement worktable. Try things,
    prototype, make a mess. Get it to about 80%: clear enough to react to.
  </Step>
  <Step title="Critique in Review">
    Drop it into **Review**, the weekly board where each page is labeled by date. The
    team reviews the week's work together and leaves feedback.
  </Step>
  <Step title="Apply feedback in Workboard">
    Take the feedback back to **Workboard** and revise. This loop can run more than once:
    exploration and critique until the design is right.
  </Step>
  <Step title="Hand off in App (or Web)">
    Produce the cleaned-up, final version in **App**: organized, ready for dev. Use
    annotations and add dev notes where they help. For web deliverables, the same thing
    happens in **Web**.
  </Step>
  <Step title="Finish in the codebase">
    After it's built, follow up with the devs and get the last 5% right, often by making
    the visual edits directly in the codebase. See [Philosophy](/handbook/philosophy).
  </Step>
</Steps>

<Callout type="tip" title="Which file am I in?">
  **Workboard** is explore. **Review** is critique (dated, weekly). **App** and **Web**
  are the clean, annotated handoff. When a Workboard passes about 100 pages it gets
  archived and the next one is started. See
  [Ideaboard and the numbering convention](/handbook/files).
</Callout>
